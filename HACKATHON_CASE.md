# Кейс хакатона: Автоматическое обнаружение персональных данных в корпоративных файловых хранилищах

## Важное примечание о датасете

Документы, специально включенные в тестовый датасет как примеры файлов, содержащих персональные данные, являются полностью вымышленными и синтетически сгенерированными исключительно для целей хакатона. Такие данные не относятся к реальным физическим лицам, не используются в операционной деятельности и не отражают сведения о существующих сотрудниках, клиентах или иных субъектах персональных данных. Любые совпадения с реальными лицами, адресами, телефонами, документами и иными идентификаторами являются случайными.

Остальные материалы датасета также носят учебно-демонстрационный характер и представляют собой либо синтетически созданные данные, либо материалы, полученные из открытых общедоступных источников. Датасет предоставляется исключительно для исследовательских и образовательных целей в рамках хакатона.

## Описание проблемы

В ходе внутреннего аудита информационной безопасности крупной компании было обнаружено, что за годы работы на корпоративных файловых серверах накопилось огромное количество файлов различного формата, которые хранятся без четкой структуры и классификации. Файлы разбросаны по различным директориям без единой системы организации.

Проблема заключается в том, что среди этих файлов могут находиться документы, содержащие персональные данные (ПДн) граждан, обработка которых регламентируется Федеральным законом № 152-ФЗ «О персональных данных». Несоблюдение требований законодательства в области защиты персональных данных может привести к серьезным штрафам и репутационным рискам для компании.

Ручной анализ таких объемов данных невозможен из-за масштаба задачи. Требуется автоматизированное решение, которое сможет:

1. Просканировать файловое хранилище со смешанной структурой файлов
2. Определить, какие файлы или группы файлов содержат персональные данные
3. Классифицировать найденные ПДн по категориям в соответствии с российским законодательством
4. Сформировать отчет о результатах анализа

## Постановка задачи

Участникам хакатона предлагается разработать программное решение (скрипт или приложение), которое:

### Основные требования:

1. **Сканирование файловой структуры**: Решение должно рекурсивно обходить указанную директорию и анализировать файлы различных форматов:
   - Структурированные данные: CSV, JSON, Parquet
   - Документы: PDF, DOC, DOCX, RTF, XLS
   - Веб-контент: HTML
   - Изображения: TIF, JPEG, PNG, GIF
   - Видео: MP4

2. **Извлечение текстовых данных**: Для файлов разных форматов необходимо реализовать механизмы извлечения текста:
   - Для структурированных форматов (CSV, JSON, Parquet) — парсинг структуры данных
   - Для документов (PDF, DOC, DOCX) — извлечение текста из документов
   - Для изображений — использование OCR (опционально, для повышения качества)
   - Для HTML — извлечение текстового контента

3. **Обнаружение персональных данных**: Решение должно выявлять следующие категории ПДн в соответствии с требованиями 152-ФЗ:

   - **Обычные персональные данные**:
     - ФИО (фамилия, имя, отчество)
     - Контактная информация (телефоны, email-адреса)
     - Дата и место рождения
     - Адреса проживания/регистрации
   
   - **Государственные идентификаторы**:
     - Паспортные данные РФ (серия, номер)
     - СНИЛС
     - ИНН (физических и юридических лиц)
     - Водительское удостоверение
     - MRZ (машиносчитываемая зона документов)
   
   - **Платежная информация**:
     - Номера банковских карт (с проверкой алгоритма Луна)
     - Банковские счета и БИК
     - CVV-коды
   
   - **Биометрические данные**:
     - Упоминания биометрических характеристик (отпечатки пальцев, радужная оболочка глаза, голосовые образцы и т.п.)
   
   - **Специальные категории ПДн**:
     - Данные о состоянии здоровья
     - Религиозные и политические убеждения
     - Расовая и национальная принадлежность
     - Иная информация, относящаяся к специальным категориям по 152-ФЗ

4. **Классификация уровня защищенности**: На основе обнаруженных категорий ПДн решение должно определять требуемый уровень защищенности (УЗ) информационной системы:
   - **УЗ-1**: Наличие специальных категорий ПДн или биометрических данных (высокий риск)
   - **УЗ-2**: Наличие платежной информации или государственных идентификаторов в больших объемах
   - **УЗ-3**: Наличие государственных идентификаторов в небольших объемах или обычных ПДн в больших объемах
   - **УЗ-4**: Наличие только обычных ПДн в небольших объемах (базовый уровень)

5. **Формирование отчета**: Решение должно генерировать структурированный отчет, содержащий:
   - Путь к файлу
   - Список обнаруженных категорий ПДн
   - Количество найденных экземпляров каждой категории
   - Определенный уровень защищенности (УЗ)
   - Формат файла
   - Рекомендации по обработке (опционально)

### Формат выходных данных

Отчет должен быть представлен в одном из следующих форматов:
- CSV файл с колонками: `путь`, `категории_ПДн`, `количество_находок`, `УЗ`, `формат_файла`
- JSON файл со структурированными данными
- Markdown отчет с визуализацией результатов

## Структура тестового датасета

Тестовый датасет представляет собой смешанную файловую структуру, имитирующую корпоративное хранилище. Датасет включает более **3000 файлов** различных форматов общим объемом около **3 ГБ**, размещенных в следующих директориях:

- **dataset** — TIF файлы со скриншотами документов (150 файлов, 18 МБ)
- **dataset0** — веб-кеш LiveJournal (HTML файлы и изображения, 1316 файлов, 476 МБ)
- **dataset1** — PDF документы, скачанные с сайта РБК (38 файлов, 40 МБ)
- **dataset2** — PDF документы, скачанные с сайта VK (21 файл, 20 МБ)
- **dataset3** — PDF документы, скачанные с сайта ЮФУ (1782 файла, 2.2 ГБ)
- **dataset4** — TIF файлы и видео (MP4) со скриншотами личных документов (23 файла, 74 МБ)
- **dataset5** — структурированные данные в формате CSV (6 файлов, 5.3 МБ, содержит десятки тысяч записей: клиенты, логистика, продукты, продажи, магазины)
- **dataset6** — структурированные данные в форматах Parquet, CSV и JSON (6 файлов, 18 МБ, содержит более 100 тысяч записей: клиенты, компании, физические лица, подписчики)

**Особенности датасета:**
- Разнообразие форматов файлов (более 10 различных типов)
- Смешанная структура данных (структурированные и неструктурированные)
- Большие объемы данных в структурированных файлах (десятки и сотни тысяч записей)
- Наличие документов в изображениях, требующих OCR
- Файлы различных размеров (от нескольких килобайт до десятков мегабайт)

**Важно**: Файлы в датасете будут перемешаны и размещены в общей директории для имитации реальной ситуации с неорганизованным хранилищем. В реальных корпоративных условиях объем данных может быть в десятки и сотни раз больше, но тестовый датасет достаточен для проверки корректности и производительности решения.

## Критерии оценки решений

Решения участников будут оцениваться по следующим критериям:

1. **Точность обнаружения ПДн** (40%):
   - Полнота выявления всех категорий персональных данных
   - Минимизация ложноположительных срабатываний
   - Корректность валидации обнаруженных данных (например, проверка алгоритмом Луна для банковских карт)

2. **Качество классификации** (25%):
   - Правильность определения уровня защищенности (УЗ)
   - Соответствие классификации требованиям 152-ФЗ

3. **Производительность и масштабируемость** (20%):
   - Скорость обработки файлов различных форматов
   - Эффективность использования ресурсов (память, CPU)
   - Возможность обработки больших файлов и больших объемов структурированных данных
   - Оптимизация обработки изображений и OCR (при использовании)
   - Способность решения масштабироваться на большие объемы данных

4. **Качество отчета** (10%):
   - Структурированность и читаемость отчета
   - Полнота представленной информации
   - Удобство для дальнейшего анализа

5. **Техническая реализация** (5%):
   - Качество кода
   - Обработка ошибок
   - Документация решения

## Технические требования

- Язык программирования: Python 3.7+ (рекомендуется)
- Разрешается использование любых открытых библиотек
- Решение должно быть запускаемым из командной строки
- Решение должно обрабатывать ошибки чтения файлов (поврежденные файлы, неподдерживаемые форматы)
- Решение не должно требовать ручного вмешательства в процессе работы

## Этические и правовые аспекты

Несмотря на синтетический и учебный характер датасета, от участников ожидается профессиональный подход к обработке информации и представлению результатов:

- В отчетах не следует сохранять или воспроизводить полные исходные значения обнаруженных сущностей; рекомендуется использовать маскирование, хеширование или частичное сокрытие значений.
- Решение должно соответствовать принципам конфиденциальности и безопасной обработки данных.
- При обнаружении ПДн или сущностей, похожих на ПДн, решение должно фокусироваться на факте наличия данных и их категории, а не на избыточном раскрытии конкретных значений.
- При работе с материалами из открытых источников участники должны соблюдать условия их использования и избегать ненужного распространения содержимого вне рамок хакатона.

## Дополнительные рекомендации

- Рекомендуется использовать регулярные выражения для поиска паттернов ПДн
- Для валидации данных (номера карт, СНИЛС, ИНН) следует использовать соответствующие алгоритмы проверки
- Для обработки изображений можно использовать библиотеки OCR (Tesseract, EasyOCR)
- Для обработки PDF рекомендуется использовать несколько библиотек для повышения качества извлечения текста
- Стоит учитывать различные форматы представления одних и тех же данных (например, телефонные номера могут быть записаны по-разному)
